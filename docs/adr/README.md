# Architecture Decision Records (ADR)

本目录包含 Wordland 项目的重要架构决策记录。

---

## 概述

ADR（Architecture Decision Record）记录重要的架构决策，包括：
- 决策的背景和上下文
- 考虑的替代方案
- 为什么选择这个方案
- 决策的后果

### 为什么需要ADR？

1. **知识共享**: 新团队成员可以快速了解历史决策
2. **决策追溯**: 为什么当初做出这个选择？
3. **避免重复讨论**: 防止同样的争论反复出现
4. **架构演化**: 记录架构如何随时间变化

## ADR 状态

### 状态定义

| 状态 | 说明 | 图标 |
|------|------|------|
| 提议 (Proposed) | 待审核的决策 | 📝 |
| 采纳 (Accepted) | 已接受的决策 | ✅ |
| 废弃 (Deprecated) | 不再推荐但仍在使用 | ⚠️ |
| 替代 (Superseded) | 已被新ADR替代 | 🔄 |

### 状态流转

```
📝 提议 → ✅ 采纳 → ⚠️ 废弃 → 🔄 替代
          ↓
       (最终归档)
```

## ADR 流程

### 1. 提议阶段

**触发条件**:
- 重大架构变更
- 技术栈变更
- 影响多个模块的决策
- 有争议的技术选择

**创建步骤**:

1. **获取ADR编号**
   ```bash
   # 查看当前最大编号
   ls -v docs/adr/*.md | tail -1

   # 下一个编号 = 最大编号 + 1
   # 例如: 最大是002，下一个是003
   ```

2. **创建ADR文件**
   ```bash
   # 使用模板创建
   cp docs/adr/TEMPLATE.md docs/adr/003-decision-title.md
   ```

3. **填写内容**
   - 清晰描述问题
   - 列出所有替代方案
   - 分析每个方案的优缺点
   - 说明选择理由

### 2. 审核阶段

**审核流程**:

1. **团队审核**
   - 在团队会议上讨论
   - 收集反馈意见
   - 修订ADR内容

2. **架构师批准**
   - 架构师审核技术合理性
   - 确认与现有架构一致
   - 批准或要求修订

3. **状态更新**
   - 提议 → 采纳（如批准）
   - 提议 → 修订（如需修改）

**审核检查清单**:

- [ ] 问题描述清晰
- [ ] 替代方案完整
- [ ] 影响分析充分
- [ ] 实施计划可行
- [ ] 相关文档链接正确

### 3. 采纳阶段

**采纳后操作**:

1. **更新索引**
   ```markdown
   ### 003: [决策标题]
   - **状态**: 已采纳
   - **日期**: YYYY-MM-DD
   - **原因**: [简要说明]
   - [查看详情](./003-decision-title.md)
   ```

2. **实施决策**
   - 按照实施计划执行
   - 记录实施过程中的问题
   - 更新实施状态

3. **更新相关文档**
   - CLAUDE.md（如架构变更）
   - 开发规范（如编码标准变更）
   - 其他相关ADR

### 4. 维护阶段

**定期审查**:

- 每季度审查一次
- 检查决策是否仍然有效
- 更新过时信息

**状态变更**:

```
✅ 采纳 → ⚠️ 废弃 (不再推荐但仍在使用)
✅ 采纳 → 🔄 替代 (已被新ADR替代)
```

**废弃条件**:
- 技术栈已更新
- 新方案明显更优
- 原决策引入新问题

### 现有ADR摘要

#### ADR 001: 使用 Service Locator

**问题**: Hilt在真机上有兼容性问题

**决策**: 使用Service Locator替代Hilt Activity注入

**关键影响**:
- ✅ 真机和模拟器行为一致
- ✅ 不依赖Hilt代码生成
- ❌ 失去编译时依赖检查

[查看详情](./001-use-service-locator.md)

#### ADR 002: Hilt兼容性方案

**问题**: Hilt 2.48 JavaPoet兼容性错误

**决策**: 保留Hilt注解，使用Service Locator获取ViewModel

**关键影响**:
- ✅ 解决兼容性问题
- ✅ 保持未来迁移灵活性
- ❌ 混合模式增加复杂度

[查看详情](./002-hilt-compatibility.md)

#### ADR 003: Screenshot Testing Approach

**问题**: Paparazzi 截图测试工具与 JDK 17 不兼容

**决策**: 采用 Compose Testing + ADB Screenshot Scripts 混合方案

**关键影响**:
- ✅ 与 JDK 17 和 JaCoCo 完全兼容
- ✅ 真机截图更准确反映实际渲染
- ⚠️ 速度较慢（真机测试）
- ⚠️ 需要人工对比截图

[查看详情](./003-screenshot-testing-approach.md)

## 常见问题

### Q1: 什么时候需要创建ADR？

**需要创建ADR的情况**:
- 重大架构变更（如更换DI框架）
- 技术栈变更（如迁移到Kotlin Multiplatform）
- 影响多个模块的决策
- 有争议的技术选择
- 引入新的模式或实践

**不需要创建ADR的情况**:
- 小的代码重构
- Bug修复
- 单一模块内的实现细节
- 临时解决方案

### Q2: ADR需要多详细？

**原则**: 足以让一年后的新团队成员理解

**最低要求**:
- 清晰的问题描述
- 至少2个替代方案
- 选择理由
- 影响分析

### Q3: ADR可以修改吗？

**可以修改**:
- 错别字和格式
- 补充实施细节
- 更新状态

**不应该修改**:
- 原始决策内容
- 选择理由（创建新ADR替代）

### Q4: 如何处理错误的ADR决策？

1. **创建新ADR替代**
2. **记录错误原因**
3. **原ADR标记为"已替代"**
4. **在经验教训中记录**
