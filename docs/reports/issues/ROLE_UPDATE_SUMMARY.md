## 🔍 Lessons Learned 总结

### 技术层面

#### Lesson 1: 理解框架的依赖关系
**问题**: 不理解 Hilt 的全局依赖链，导致移除注解后崩溃

**Hilt 依赖链**:
```
@HiltAndroidApp (Application)
    ↓ 必需
@AndroidEntryPoint (Activity)
    ↓ 必需
hiltViewModel<VM>() (Screen)
    ↓
Hilt 自动创建 ViewModel
```

**关键认知**:
- Hilt 是**全有或全无**的设计
- `hiltViewModel()` 严格要求 `@AndroidEntryPoint`
- ❌ 不能"局部移除"Hilt
- ✅ 要么完全使用 Hilt，要么完全不使用

#### Lesson 2: 真机 vs 模拟器差异
**关键差异**:
| 特性 | 模拟器 | 真机 | 风险 |
|------|--------|------|------|
| Hilt 代码生成 | ✅ 稳定 | ⚠️ 可能失败 | 🔴 高 |
| 文件 I/O | ⚡ 快 | 🐢 慢 | 🟡 中 |
| 内存限制 | 🟧 无限 | 🟣 严格 | 🟡 中 |

**关键认知**:
- 模拟器成功 ≠ 真机成功
- **真机测试是必需的**，不是可选项
- 要在真机上验证依赖注入

#### Lesson 3: 修复策略要系统化
**错误方式**:
```
发现问题 → 快速修复 → 验证错误消失 → 完成 ❌
```

**正确方式**:
```
发现问题
  ↓
完整分析（识别所有使用点）
  ↓
设计方案（评估影响范围）
  ↓
实施方案
  ↓
全面测试（不只是错误点）
  ↓
文档化（记录原因和方案）✅
```

### 流程层面

#### Lesson 4: 真机测试流程缺失
**之前的流程**:
```
开发 → 模拟器测试 → 构建 → 发布 ❌
```

**现在的流程**:
```
开发 → 模拟器测试 → 真机测试 → 构建 → 发布 ✅
                    ⬆
                  必需环节
```

#### Lesson 5: 完成标准不完整
**之前的标准**:
```yaml
完成:
  - 模拟器测试通过 ✅
  - 功能正常 ✅
```

**现在的标准**:
```yaml
完成:
  - 单元测试通过 (>80% 覆盖率) ✅
  - 模拟器测试通过 ✅
  - 真机测试通过 ⚠️ 新增
  - 核心功能验证 ✅
  - 无崩溃和内存泄漏 ✅
  - 文档更新（ADR, RCA）⚠️ 新增
```

## 📝 角色定义更新详情

### Android Architect 更新

#### 新增职责
1. **理解框架的完整依赖链**
   - 特别是 Hilt/Dagger 的全局依赖关系
   - 评估架构变更的影响范围

2. **记录架构决策（ADR）**
   - 为什么选择/放弃某个技术
   - 已知限制和风险
   - 真机 vs 模拟器差异
   - 依赖关系要求
   - 迁移路径

3. **定义真机测试流程**
   - 真机测试不是可选项，是必需的
   - 定义真机测试标准
   - 建立质量门禁

4. **建立 Root Cause Analysis 流程**
   - 问题诊断文档化
   - 防止问题复发的措施

#### 新增指南章节
**真机测试与问题诊断指南**:
- 真机测试流程（开发阶段、集成阶段、发布阶段）
- 框架依赖链理解
- 真机 vs 模拟器差异对比
- 问题诊断流程（6 个步骤）
- ADR 模板
- Root Cause Analysis 模板
- 常见问题 FAQ

### Android Engineer 更新

#### 新增职责
1. **理解依赖注入机制**
   - Hilt/Service Locator/Koin 的工作原理
   - 确保 ViewModel 获取方式与依赖注入配置一致

2. **真机测试验证** ⚠️ 关键职责
   - 在真机上验证代码（不只是模拟器）
   - 测试完整功能流程（不只是修复的错误点）
   - 检查依赖注入是否正常工作
   - 验证真机特有功能
   - 性能测试

3. **问题诊断与文档化**
   - 完整分析问题（不是快速修复）
   - 识别所有影响点（不只是错误点）
   - 编写 Root Cause Analysis
   - 更新架构决策记录（ADR）

#### 新增指南章节
**真机测试验证指南**:
- 真机测试清单（开发者自测、完整测试）
- 依赖注入验证步骤
- 完整性问题修复流程
- 问题诊断模板（8 个步骤）
- 常见陷阱（3 个）
- 完成标准（代码、模拟器、真机、文档）

## 🎯 防止复发措施

### 技术措施
1. **Service Locator 作为标准模式**
   - 不依赖 Hilt Activity 级注入
   - 真机和模拟器行为一致
   - 更容易调试和理解

2. **保留兼容性设计**
   - ViewModels 保留 `@HiltViewModel` 注解
   - 支持未来迁移到其他 DI 框架

3. **添加真机测试脚本**
   - 自动化真机测试流程
   - 检查清单

### 流程措施
1. **更新开发流程**
   ```
   开发 → 模拟器测试 → 真机测试 → 构建 → 发布
                       ⬆
                     必需
   ```

2. **定义"完成"标准**
   - 单元测试 >80% 覆盖率
   - 模拟器测试通过
   - **真机测试通过** ⚠️
   - 核心功能验证
   - 无崩溃和内存泄漏
   - **文档更新** ⚠️

3. **建立 Architecture Decision Record**
   - 记录所有架构决策
   - 包括真机 vs 模拟器差异
   - 已知限制和风险

### 组织措施
1. **知识分享**
   - 每个 RCA 都要分享
   - 定期 Lessons Learned Session
   - 建立 FAQ 和 Troubleshooting Guide

2. **质量文化**
   - "真机测试不是可选项，是必选项"
   - "修复要完整，不能只治标"
   - "文档是为了未来的自己"
