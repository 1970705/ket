## Overview

作为Android架构师，你负责推动以下质量改进活动，确保代码质量和项目质量持续提升。

---

## ⚡ 立即行动（本周）- P0优先级

### 1. 建立Code Review Checklist ⚠️ CRITICAL

**目标**: 确保每次Code Review不遗漏关键检查点

**创建文档**: `docs/development/CODE_REVIEW_CHECKLIST.md`

**检查清单必须包含**:
- [ ] **架构验证**: DI/框架变更的完整职责已实现
- [ ] **逻辑验证**: 命名与实现一致性（如unlock函数必须解锁）
- [ ] **数据验证**: 无硬编码数据，使用ViewModel
- [ ] **测试验证**: 新代码有对应测试
- [ ] **真机验证**: 在真机上测试关键流程
- [ ] **文档验证**: ADR已创建（如涉及架构变更）

**使用方式**:
- 每个PR必须通过检查清单
- 架构师必须签字批准
- 不符合清单的PR必须修改

---

### 2. 建立发布前测试流程 ⚠️ CRITICAL

**目标**: 防止有bug的代码到达生产环境

**创建脚本**: `scripts/release/pre_release_check.sh`

**必须通过的测试**:
```bash
# 1. 单元测试
./gradlew test

# 2. 首次启动测试（空数据库）
./scripts/test/test_first_launch.sh

# 3. 真机全新安装测试 ⚠️ MANDATORY
./scripts/test/test_real_device_clean_install.sh

# 4. 带崩溃检测的导航测试
./scripts/test/test_navigation_with_crash_detection.sh
```

**规则**:
- ⛔ 任一项失败 = 阻止发布
- ⛔ 真机测试不能跳过
- ⛔ 首次启动测试必须通过

---

### 3. 定义架构变更验证流程

**目标**: 确保架构重构不遗漏职责

**验证流程**:
```
架构变更前:
1. 列出当前框架的所有职责
2. 识别哪些依赖需要手动处理
3. 创建迁移检查清单

架构变更中:
4. 逐项手动实现
5. 逐项验证完成
6. 单元测试验证

架构变更后:
7. 在空数据库环境测试
8. 真机全新安装测试
9. 创建ADR文档
10. Code Review检查清单验证
```

**输出**: ADR文档（`docs/adr/XXX-architecture-change.md`）

---

### 4. 创建质量度量标准

**目标**: 建立量化的质量目标

**度量指标**:
- 单元测试覆盖率: 目标 > 80%
- 集成测试覆盖率: 目标 > 60%
- 零编译警告
- 零Lint错误
- 零已知Bug

## 📈 中期目标（1个月）- P1优先级

**目标**: 自动化质量检查，持续集成

**设计任务**:

- 设置测试覆盖率报告（JaCoCo）

**验证点**:
- [ ] 提交代码自动运行测试
- [ ] PR显示测试结果
- [ ] 测试失败阻止合并
- [ ] 覆盖率报告自动生成

### 2. 选择静态分析工具

**目标**: 自动检测代码质量问题

**候选工具**:
- **Detekt**: Kotlin静态代码分析
- **ktlint**: Kotlin代码风格检查
- **Spotless**: 代码格式化
- **Android Lint**: Android特定检查

**配置要求**:
- 自定义规则：
  - 禁止硬编码数据（levelInfo等）
  - 检测TODO/FIXME未完成
  - 检测命名不一致
- 集成到Gradle构建
- 集成到CI/CD

**验证点**:
- [ ] 可以检测硬编码数据
- [ ] 可以检测命名不一致
- [ ] 集成到CI/CD
- [ ] 团队接受并使用

## 🚀 长期目标（3个月）- P2优先级

### 1. 引入测试驱动开发(TDD)流程

**目标**: 提升代码质量和测试覆盖率

**TDD流程定义**:
```
1. 🔴 红灯: 写失败的测试
   - 定义接口/行为
   - 测试失败（编译失败或断言失败）

2. 🟢 绿灯: 写最小实现
   - 实现代码使测试通过
   - 不添加额外功能

3. 🔵 重构: 优化代码
   - 改进设计
   - 保持测试通过
   - 消除坏味道

4. 🔁 循环: 重复
   - 添加下一个测试
   - 持续迭代
```

**适用范围**:
- ✅ 新的UseCase必须TDD
- ✅ 新的算法必须TDD
- ✅ 新的业务逻辑必须TDD
- ⚠️ UI层可选（可以后置测试）

**验证点**:
- [ ] 团队接受TDD流程
- [ ] 新代码80%使用TDD
- [ ] 测试覆盖率提升到85%+

---

### 2. 建立质量门禁

**目标**: 强制质量标准，自动化质量保证

**质量门禁规则**:
```yaml
# 自动化检查（阻止合并）
pre-commit:
  - 单元测试必须通过
  - 代码覆盖率不能下降
  - Lint必须通过
  - 静态分析必须通过

# 手动检查（架构师批准）
code_review:
  - Code Review Checklist完成
  - 架构师签字批准
  - 无"TODO"或"FIXME"未完成

# 发布前检查（架构师+测试工程师）
pre_release:
  - 所有测试套件通过
  - 真机测试通过（至少2台设备）
  - 性能测试通过
  - 零已知Bug
```

**实施方式**:
- Git Hooks自动检查
- GitHub Actions CI检查
- 架构师手动批准
- 发布前测试报告

**验证点**:
- [ ] 质量门禁自动化
- [ ] 团队遵守规则
- [ ] Bug逃逸率降低到<5%

---

### 3. 培养测试文化

**目标**: 让测试成为团队DNA，而不是负担

**文化活动**:
- **测试分享会**: 每周分享测试技巧和最佳实践
- **Bug回顾会**: 每个Bug都要做RCA，讨论如何预防
- **测试黑客松**: 每季度举办，奖励发现Bug的测试
- **测试培训**: 新人入职必须接受测试培训

**验证点**:
- [ ] 团队成员主动写测试
- [ ] Code Review时关注测试
- [ ] 测试覆盖率持续上升
- [ ] 零"测试是额外工作"言论
